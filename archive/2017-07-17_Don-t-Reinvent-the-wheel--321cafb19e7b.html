Don’t Reinvent the wheel!

I know this is a common phrase that is heard quite a bit in the engineering and software engineering community, but please don’t reinvent the wheel!

I finally completed the first dynamic full-stack back-end project for FreeCodeCamp. This was a voting application.

The main issue was that the coding template that was given was super bare bones. This probably added a 30–50% implementation overhead on the whole entire project.

Here are some general guidelines when you start a project to avoid solving common problems. This allows you to simply work on the complicated issues and link existing solutions together.

  • Search for the abstracted version of the problem (i.e the wheel)
  • See if there is a framework or software that alleviates the difficulty (i.e engine)
  • divide the problem into a series of smaller common problems (i.e gears)

Search for the abstraction

Abstraction is the ability to establish a layer of complexity that the engineer or programmer can work with.

One good example of abstraction is programming languages. As computers have evolved, so has the creation of software.

Early computer programs were written with lower level languages such as Assembly. This placed a limit to how much could be built over a period of time.

As a result, tools were built to go beyond the complexity of these early programming languages.

Look for frameworks or software that makes it easier

This is an extension of the previous statement. Major forms of abstraction include frameworks and software.

Frameworks provide a well-tested foundation to build applications and solutions on. This is better then having to build the foundation all by yourself.

For example, some good front-end frameworks for web development are React and Angular.

Divide and Conquer

If the problem seems to complex, that probably because it is made up of multiple problems. The solution to this is by splitting the complex problem into smaller simple problems.

The problem looks less daunting. All you need to do now is construct all the gears and link them together. A bunch of small gears will take less time to produce then one large complicated one.

Conclusion

The main goal is to make sure to explore other options before diving headfirst into unneeded complexity.
